Boundary acquired Graphdat.com in order to more broadly expand its existing network flow monitoring service with more general purpose server metric monitoring while keeping one second resolution on both. After acquisition I worked closely with engineering to integrate Graphdat realtime graphs onto existing dashboards and also to add scalability for customer growth. In terms of scalability we moved the existing explicit cluster worker mapping scheme to a consistent hash based scheme (Guava), Zookeeper presence, with a finer grained tuple to better balance load. Internal to the data storage worker I used a similar hashing scheme to partition open files among worker threads to better drop contention. At the same time I added skip lists to time series host name lookups and utilized memory mapped I/O to greatly reduce system context switches and dirty page management. The end result, our production data storage nodes now run comfortably doing 1.2M inserts/updates per second with negligible CPU usage.

Graphdat.com / AlphashackCTO - July 2012-April 2014 - Seattle, WA

  • Managed all aspects of the business (investors, staffing, etc..)
  • Primary architect of the one second monitoring technology behind Graphdat.com
  • Built a browser based graphing engine offering pan, zoom, and realtime views
  • Created a distributed metric data store optimized for one second data
  • Created a small footprint agent for collecting machine telemetry data
  • Acquired by Boundary.com

The idea behind Graphdat.com came about while working at C4 and experiencing extremely high gameday web traffic. We had some very expensive monitoring tools which could only refresh graphs every twenty seconds. Our goal with Graphdat.com was to provide a cloud based service, to monitor any metric, at one second resolution, easily and cheaply. I designed and built the majority of the system which was composed of three main parts, a metric data collection agent, fast backend data storage, and an interactive graphing browser UI. For the lowest overhead memory and CPU the data collection agent was written in C (open source libsigar for metric reading). Because we pre-aggregate data (similar to OLAP cube rollups) each point generated many other points so the data store needed to be very efficient. Our testing suggested we could get the best performance if I wrote a custom built data store in C. Finally, for the UI, at the time we found client based vector graphics using SVG were sluggish so I instead built a canvas based interactive graph which used a sparse time range data cache to optimize server communication. I worked closely with one other engineer to release Graphdat.com which was then acquired after approximately 2 years of development.

The greatest challenge of C4 was managing expectation for a large, non-technical, population of stakeholders while simultaneously assisting with the technical challenges related to websites which would change traffic profiles dramatically during game days. For example, unique visitors during a weekday may be several thousand but on a weekend with matches we may see 4M unique visitors. This unique performance curve in conjunction with our general purpose CMS (DotNetNuke) which was serving some rather complicated sport summary data made for some difficult performance scenarios. I had to problem solve at the database tier to find slow queries, remove xml transforms, stop intermediate sorts which turned reads to writes, and insure caching. From there I proposed and built a centralized shared caching solution which would become an early predecessor to REDIS open source. It was during this time that the beginning of the idea for Graphdat.com came about. C4 was acquired by Photon Group Pty Ltd.
